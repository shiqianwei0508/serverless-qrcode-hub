## 用户需求

为二维码后台管理面板新增条目模糊搜索功能。

## 产品概述

在后台「全部」条目列表顶部增加一个搜索框，支持按「条目名称」与「短链名」进行实时模糊匹配过滤。搜索采用后端 SQL 模糊查询，保证跨分页结果准确。

## 核心功能

- 「全部」列表顶部新增搜索输入框，带放大镜图标，占位提示「搜索条目名称 / 短链名…」
- 实时输入触发（约 300ms 防抖），输入即按 条目名称(name) 或 短链名(path) 子串模糊匹配，并重置到第 1 页重新加载
- 搜索仅作用于「全部」视图；点击「即将过期 / 已过期」切换时清空搜索框与关键词，避免搜索状态残留
- 后端 /api/mappings 接收 search 参数，对 name 与 path 做 LIKE %关键词% 匹配，分页与总数统计随之生效
- 置顶排序、表头排序、分页、每页大小等现有行为在搜索后仍正常工作

## 技术栈与约束

- 运行时：Cloudflare Workers（无构建步骤），后端单文件 `index.js`
- 数据库：Cloudflare D1（SQLite 兼容），查询走 `DB.prepare(...).bind(...).all()` 参数化
- 前端：`dist/admin.html` 直接作为静态资源服务（wrangler.toml `assets.directory = "./dist"`），原生 JS + Tailwind/DaisyUI，无打包
- 搜索范围：仅「全部」列表；触发：实时防抖

## 实现方案

采用后端 LIKE 模糊查询（而非前端纯过滤），原因：列表分页加载，前端仅持有当前页数据，纯前端过滤会遗漏其他页结果；后端过滤才能保证跨页完整。D1/SQLite 的 `LIKE` 对 ASCII 默认大小写不敏感，中文按子串匹配，满足「模糊」需求，且用参数化绑定天然防注入。

关键决策：

1. `listMappings(page, pageSize, search = '')` 增加可选 search 参数；仅当 search 非空时向 `filtered_mappings` CTE 追加 `AND (name LIKE ? OR path LIKE ?)`。因 COUNT 子查询引用同一 CTE，总数也会随搜索自动收敛，分页正确。
2. 路由 `/api/mappings` 读取 `search = params.get('search') || ''`，透传；做一次长度裁剪（如 `search.slice(0, 64)`）降低异常输入风险。
3. 前端新增 `searchInput` 与状态 `searchKeyword`，`loadMappings` 在 searchKeyword 非空时拼接 `&search=`，空时保持原 URL 行为。
4. 防抖 300ms：输入后重设 `currentPage=1` 再 `loadMappings()`；切换「即将过期/已过期」时清空输入框与关键词（这两个视图不改搜索，避免用户困惑）。

性能与可靠性：LIKE 对全表扫描在无索引时成本 O(n)，但条目规模小（二维码活码，通常数百级），可忽略；`name` 为可空列，`name LIKE ?` 对 NULL 行自动不命中，由 `path`（必有值）提供兜底，逻辑正确。后端复用现有 `listMappings` 结构与 `banPath` 占位符拼接模式，不引入新架构。

## 实现注意

- bind 参数顺序：`...banPath, '%'+search+'%', '%'+search+'%', pageSize, offset`，需与 SQL 中占位符数量严格对应，避免参数错位。
- 前端清空逻辑要同时重置 `searchInput.value` 与 `searchKeyword`，并在 showExpiringBtn/showExpiredBtn 事件里调用，保持与「全部」视图互不干扰。
- 维持现有表头排序（基于已加载 allMappings 前端排序）不变，搜索（后端过滤）与排序（前端排序）可叠加生效。

## 关键代码要点

- `listMappings` 条件拼接（仅当 search 非空）：
- SQL：`WHERE path NOT IN (?) AND (name LIKE ? OR path LIKE ?)`
- bind：`...banPath, \`%${search}%\`, \`%${search}%\`, pageSize, offset`
- 路由：`const search = (params.get('search') || '').slice(0, 64);` 后调用 `listMappings(page, pageSize, search)`
